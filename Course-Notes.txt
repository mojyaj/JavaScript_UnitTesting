26. Testing For Thrown Errors & Error Messages

		Checks that a function throws an error, and optionally verifies what kind of error it is.
		*can take 1 optional argument
		Method Definition:
			(method) Vi.JestAssertion<() => void>.toThrow(expected?: string | RegExp | (new (...args: any[]) => any) | Error | undefined): void

50. Testing Asynchronous Code With Callbacks

Original problem is explained better in this lecture

Talking Points
	- '.toBe()' & '.toEqual()' can throw error and Vitest will show reason for failure
	- Avoid false positives in 'callback-based' async tests using try/catch

Problem
	- False Positive
	- Synchronous test code will finish before Asynchronous test code
		- Assertions/expect() inside of asynchronous functions will not get the chance to execute
		- ** A test is marked as Passed if there is no assertion/expect() **
Solution/Fix
	- Pass a Callback function as an argument into anonymous function 'it()' or 'test()'
	- Callback function will be called before test execution is complete	allowing
		assertions/expect() to be properly evaluated 
	- Call inside of async functions
		- This implies that testable async functions MUST define a callback function in its parameters
	- Example, 				
		it('test case', (doneFn) => {
							^^^^^^^
			const input = 'test@test.com';

			generateToken(input, (err, token) => {
				expect(token).toBe(2);

				doneFn();  THIS STILL FAILS -> Error: Timeout
				^^^^^^
			});
		});
	- If 'doneFn()' is not called, test will hang or timeout (FAIL)
	New Problem
		- Error: Timeout
		- Culprit: '.toBe(2)' failed and an error is thrown
		- Vitest is waiting for doneFn() and its never called
	Fix: 
		- Wrap async test code in try/catch
		- Example,
			generateToken(testUserEmail, (err, token) => {
				try {
					expect(token).toBe(2); 
					doneFn();
				}
				catch (err){
					doneFn(err);
				}
			});

52. Returning Promises In Tests

*no video

Talking Points
	- 'return' keyword signals Vitest to "wait for this 'Promise' to finish before ending the test"
	- 'return' keyword is needed whenever a test returns a 'Promise' and it does not use 'async/await' keywords

Instructors Document:

Even though the test worked as expected in the previous lecture, you should actually return 
the promise assertion in your tests:

it('should generate a token value', () => {
  const testUserEmail = 'test@test.com';
 
  return expect(generateTokenPromise(testUserEmail)).resolves.toBeDefined();
});
This guarantees that Vitest / Jest wait for the promise to be resolved.

You don't need to return when using async / await (since a function annotated with async returns 
a promise implicitly).

Instructors document:

Concurrency & Default Behavior
Even when not adding the .concurrent property / annotation, tests that are stored in different 
files are executed concurrently (i.e., in parallel). This is done by both Vitest and Jest - ensuring 
that your tests run in a short amount of time.

With .concurrent you can enforce this behavior also inside the individual files (i.e., tests that live 
in one and the same file are executed concurrently).

Concurrent execution can reduce the amount of time your tests need to execute. A downside of concurrent 
execution is, that tests that perform clashing (global) state manipulations may interfere with each other.
